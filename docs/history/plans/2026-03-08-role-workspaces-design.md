# AI Sales OS Role Workspaces Design

日期：2026-03-08

状态：待评审

范围：只设计三个角色工作台，不改其它对象页和系统页

补充范围确认：

- 不改造现有 `/home`
- 新增三个一级页面
- 在左侧导航注册三个新入口

相关依据：
- `AGENTS.md`
- `ai_sales_os_frontend_product_plan.md`

## 1. 设计目标

本轮不再继续把首页做成“更好看的 CRM 总览”。

本轮设计目标是新增三个角色工作台入口：

1. CEO 工作台
2. 销售主管工作台
3. 一线销售工作台

这三个工作台都必须遵守同一个原则：

```text
用户先看到“现在最重要的事和决策点”，
再决定是否下钻到 Deal / Meeting / Pipeline / Revenue 等对象页查看证据和细节。
```

因此：

- CRM、会议、邮件、聊天、Pipeline 表格都退到下层
- 上层入口不暴露大而全列表
- 上层优先暴露判断、风险、动作、批准点、经营影响
- Agent 不是边栏附属物，而是每个工作台的主叙事者

---

## 2. 当前问题判断

当前项目的方向是对的，但角色首页仍然偏像“带 Agent 摘要的 CRM 仪表盘”。

核心问题不是是否有 Agent 卡片，而是顶层页面仍然主要按这些元素组织：

- KPI
- 风险列表
- 重点商机列表
- 趋势图
- 数据告警

这会带来三个问题：

1. 用户仍然在读系统，不是在和系统一起判断
2. 重要程度被视觉平铺，缺少“现在必须处理什么”的强优先级
3. CEO / 销售主管 / 一线销售看到的仍是同一套对象的不同浓度，不是不同层级的决策界面

结论：

```text
现有首页更像“对象和指标的聚合层”，
新工作台应该变成“决策与行动的编排层”。
```

---

## 3. 三种设计方向

### 方案 A：保留仪表盘结构，只加强 Agent 摘要

定义：

- 继续保留 KPI、列表、趋势图为主体
- 在顶部增加更强的 Agent Brief
- 把证据与建议动作做得更明显

优点：

- 改动最小
- 对现有页面结构兼容度高
- 上线速度快

缺点：

- 本质仍像 CRM/BI
- Agent 仍然是解释器，不是主导者
- 很难体现“代 CEO 做经营判断”的产品姿态

适用性判断：

- 不推荐作为目标形态
- 只适合作为过渡方案

### 方案 B：决策卡片优先的角色工作台

定义：

- 每个角色首页都以“判断卡 / 决策卡 / 介入卡 / 动作卡”为主
- KPI、趋势、对象列表退居二级信息
- Agent 先给结论，再给影响，再给建议，再给证据入口

优点：

- 最符合 Agent-first
- 能清晰区分 CEO、主管、销售三个视角
- 更容易把系统从 CRM 升级成 revenue management / decision system

缺点：

- 需要重新定义顶层信息结构
- 对卡片语义和优先级设计要求更高
- 后续实现时需要补更强的状态模型

适用性判断：

- 推荐
- 应作为本轮设计的主方案

### 方案 C：Agent Inbox 优先的单线程工作台

定义：

- 顶层几乎不展示结构化模块
- 所有角色首先进入 Agent 对话收件箱
- 系统按时间和优先级推送任务、风险和问题

优点：

- Agent 存在感最强
- 自主性最强
- 易于表达“系统主动工作”

缺点：

- 过于 chat-first
- 复杂经营判断缺少空间感和稳定视图
- 不利于用户快速横向比较多个决策项

适用性判断：

- 不适合当前阶段
- 可作为后续的 Agent Workspace 演进方向

---

## 4. 推荐结论

推荐采用：

```text
方案 B：决策卡片优先的角色工作台
```

同时吸收方案 C 的一个优点：

- 每个工作台顶部都要有 Agent 主叙事区

最终形态不是“仪表盘 + 聊天框”，而是：

```text
Agent 主叙事
↓
高优先级决策卡片
↓
系统已在推进中的动作
↓
需要用户批准/介入的事项
↓
证据与对象下钻入口
```

---

## 5. 顶层产品框架

建议把顶层工作台理解为三个脑层：

1. CEO 工作台 = `Revenue Brain`
2. 销售主管工作台 = `Pipeline Engine`
3. 一线销售工作台 = `Deal Brain + Interaction Intelligence`

对应关系：

- CEO 不是管理 Deal 列表，而是管理收入判断和资源配置
- 销售主管不是管理 CRM 表格，而是管理团队风险和介入动作
- 一线销售不是维护字段，而是推进客户承诺和下一步动作

因此顶层信息架构不再是：

- Deals
- Meetings
- Pipeline
- Revenue

而应该是：

- 现在最重要的事
- 为什么重要
- 建议决策
- 谁来处理
- 系统是否可代办
- 影响金额 / 风险 / 时间窗口
- 证据入口

对象页的角色改为：

- `Deal / Meeting / Pipeline / Revenue` 是证据页、分析页、执行页
- 三个工作台才是经营入口

## 6. 共享设计原则

### 6.1 顶层只展示决策，不展示信息堆积

每个工作台首页默认只回答四个问题：

1. 现在最重要的三到五件事是什么
2. 为什么这些事重要
3. 建议怎么做
4. 这件事需要用户批准、修改，还是系统可直接代办

### 6.2 重要事项必须卡片化

顶层核心单元建议统一为 `Decision Card`。

每张卡片至少包含：

- 一句话判断
- 影响范围
- 建议动作
- 置信度
- 数据完整度
- 最近更新时间
- 最多三条依据
- 查看证据
- 批准 / 改写 / 驳回 / 延后

### 6.3 KPI 不消失，但降级为上下文

KPI 的作用改为：

- 为判断提供规模背景
- 帮助用户理解影响量级

KPI 不应继续占据页面最强视觉焦点。

### 6.4 对象列表不作为首页主舞台

Deal 列表、会议列表、客户列表可以保留，但只能作为：

- 决策卡的展开结果
- 证据追踪入口
- 二级浏览页

### 6.5 自主性必须可见

每个工作台都必须明确区分：

- 系统已判断出的事情
- 系统建议代执行的事情
- 系统已经启动但待批准的动作
- 必须由人决定的事项

### 6.6 证据与信任状态持续可见

所有决策卡必须显示：

- 数据新鲜度
- 数据覆盖率
- 缺失项
- 低置信度提示

## 7. 共享卡片体系

### 7.1 Decision Card

用于承载一个需要判断或批准的经营事项。

适用角色：

- CEO
- 销售主管
- 一线销售

字段建议：

- `title`
- `summary`
- `impact_label`
- `impact_value`
- `recommendation`
- `confidence`
- `freshness`
- `coverage`
- `due_window`
- `owner`
- `autonomy_level`
- `approval_required`
- `evidence_refs`
- `status`

### 7.2 Intervention Card

用于承载“谁应该介入哪个问题”的管理动作。

适用角色：

- CEO
- 销售主管

字段建议：

- 介入对象
- 介入原因
- 预期改变
- 推荐介入人
- 时机窗口
- 是否已发起

### 7.3 Action Plan Card

用于承载一线销售的当前行动方案。

适用角色：

- 一线销售

字段建议：

- 当前目标
- 建议下一步
- 已准备草稿
- 阻塞点
- 需要确认项
- 关联客户承诺

### 7.4 Trust Card

用于承载“这个判断哪里不可靠”。

适用角色：

- 全角色

字段建议：

- 缺失数据源
- 影响的判断
- 风险说明
- 建议补齐动作

## 8. CEO 工作台设计

### 8.1 核心定位

CEO 工作台不是“收入总览页”，而是：

```text
公司收入经营的决策台
```

它服务的不是“查看情况”，而是：

- 识别本季度收入偏差
- 判断哪些关键 deal 会影响达成
- 决定资源、节奏、主管介入和战略优先级

### 8.2 CEO 最关心的问题

页面必须优先回答：

1. 本季度收入是否偏离目标
2. 哪三个问题最可能造成偏差
3. 哪些关键商机需要 CEO 级决策或背书
4. 哪些动作系统可以先代做，哪些必须我批准

### 8.3 页面结构

建议结构：

1. `CEO Agent Brief`
2. `Top Decisions`
3. `Revenue Exposure`
4. `Strategic Bets`
5. `Interventions In Flight`
6. `Trust And Data Integrity`

### 8.4 模块定义

#### CEO Agent Brief

作用：

- 用一句话给经营结论
- 用两到三条原因解释变化
- 给出一条本周最值得做的经营动作

内容示例：

- 本季度仍可达标，但达标路径正在收窄
- 两个大额商机的赞助人和法务状态在削弱预测置信度
- 建议本周要求主管完成对两条战略 deal 的介入闭环

#### Top Decisions

用 3 张高优先级 `Decision Card` 表达：

- 哪个判断最重要
- 如果不处理会影响多少收入
- 推荐决策是什么

这块应该是全页视觉焦点。

#### Revenue Exposure

不是传统 forecast 图，而是：

- 当前收入风险暴露额
- 造成暴露的主要因子
- 如果延迟或失单，影响如何变化

这块强调“经营后果”，不是强调“图表完整”。

#### Strategic Bets

展示最关键的 3 到 5 个 deal，但不以表格呈现。

每个战略商机只暴露：

- 当前判断
- 为什么重要
- 需要谁介入
- 是否需要 CEO 背书

#### Interventions In Flight

展示系统和团队已经启动的动作：

- 哪些主管已被建议介入
- 哪些动作在等待批准
- 哪些动作执行后还未验证结果

CEO 需要看到系统是否在主动推进，而不是只看静态风险。

#### Trust And Data Integrity

单独显示：

- 哪些收入判断不完全可信
- 是因为数据过期、缺失，还是同步失败
- 哪些结论当前不应当被过度依赖

### 8.5 CEO 不应看到什么

不应默认看到：

- 大量 deal 表格
- 长列表 CRM 字段
- 全量会议列表
- 低层级操作按钮

这些内容应全部退到下钻页。

### 8.6 CEO 成功体验

CEO 打开页面后，应该在 30 秒内完成：

1. 理解经营偏差
2. 知道影响最大的 3 个点
3. 做出批准、介入或要求跟进的决定

## 9. 销售主管工作台设计

### 9.1 核心定位

销售主管工作台不是“团队 Pipeline dashboard”，而是：

```text
团队推进与介入管理台
```

它服务的不是“看管道”，而是：

- 判断哪里推进失真
- 判断谁该被辅导
- 判断自己要介入哪个 deal
- 让系统协助发起管理动作

### 9.2 销售主管最关心的问题

页面必须优先回答：

1. 本周最需要介入的三个风险是什么
2. 哪些 deal 停滞且会影响团队结果
3. 哪些销售动作质量不够，需要辅导
4. 哪些管理动作系统已经准备好，只等我确认

### 9.3 页面结构

建议结构：

1. `Manager Agent Brief`
2. `Intervention Queue`
3. `Pipeline Reality Gap`
4. `Rep Coaching Queue`
5. `Team Commitments`
6. `Data And Sync Exceptions`

### 9.4 模块定义

#### Manager Agent Brief

给出：

- 本周团队总体判断
- 三个最重要的风险来源
- 最值得主管介入的一条管理动作

#### Intervention Queue

这是主管工作台的核心模块。

每张卡片回答：

- 哪个 deal 出了问题
- 根因是什么
- 推荐谁介入
- 介入目标是什么
- 现在是否等待主管批准

这块是主管版本的 `Decision Card + Intervention Card` 组合。

#### Pipeline Reality Gap

这块不是普通漏斗，而是“现实偏差层”。

重点表达：

- CRM 阶段与真实推进是否一致
- 后期 deal 是否缺少真实承诺
- 哪些商机表面健康、实际脆弱

这一块要体现“系统在重建 pipeline 真实性”。

#### Rep Coaching Queue

主管看到的不是泛化评分，而是：

- 哪位销售在哪个 deal 或会议上暴露出什么模式问题
- 推荐的辅导动作是什么
- 是否需要跟会、复盘、共创下一步

#### Team Commitments

把团队已经承诺的关键下一步整理成承诺视图：

- 本周必须推进的客户动作
- 是否有日期、 owner、预期结果
- 哪些承诺正在滑落

主管管理的不是字段，而是承诺兑现。

#### Data And Sync Exceptions

主管层面必须看到：

- 哪些判断因为同步失败而失真
- 哪些销售没有留下一手客户互动数据
- 哪些风险来自数据盲区而不是销售动作本身

### 9.5 销售主管不应看到什么

不应默认看到：

- 全量 deal table
- 全量会议流
- 过多 KPI 卡片
- 为了好看堆叠的趋势图

### 9.6 销售主管成功体验

销售主管打开页面后，应该在 2 分钟内完成：

1. 找到需要自己介入的 deal
2. 找到需要辅导的销售
3. 确认系统建议发起的管理动作

## 10. 一线销售工作台设计

### 10.1 核心定位

一线销售工作台不是“我的 CRM 首页”，而是：

```text
今日推进与客户承诺执行台
```

它服务的不是“更新记录”，而是：

- 明确今天先推什么
- 帮用户准备下一步动作
- 暴露当前 deal 的真实阻塞
- 让 Agent 代起草、代整理、代提醒

### 10.2 一线销售最关心的问题

页面必须优先回答：

1. 今天最该推进哪三件事
2. 每件事为什么要先做
3. Agent 已经帮我准备了什么
4. 哪些地方因为信息不足，我不该过度自信

### 10.3 页面结构

建议结构：

1. `Rep Agent Brief`
2. `Today Focus Stack`
3. `Customer Commitment Tracker`
4. `Prepared By Agent`
5. `Deal Risks To Resolve`
6. `Missing Context`

### 10.4 模块定义

#### Rep Agent Brief

顶部只说最直接的话：

- 今天先做什么
- 为什么先做
- 如果不做会卡住什么

语气应像优秀销售同事，不像系统提示。

#### Today Focus Stack

这是销售首页的核心。

每张 `Action Plan Card` 对应一个立即动作：

- 目标客户 / deal
- 需要完成的动作
- 成功标准
- 建议发送内容或会议目标
- 是否等待确认

#### Customer Commitment Tracker

销售需要看到客户承诺是否真的往前推进，而不是只看阶段。

这块建议围绕：

- 客户答应了什么
- 谁负责
- 什么时候到期
- 哪些承诺正在失效

#### Prepared By Agent

展示系统已经代准备好的内容：

- 跟进邮件草稿
- 会议 brief
- Next Step 草稿
- 异议回应要点

重点是让销售感觉系统已经替他做了一部分工作。

#### Deal Risks To Resolve

显示销售手里最需要清掉的阻塞：

- 决策人未明确
- 预算未确认
- 实施负担未解释
- 法务 / 安全异议未处理

这块要面向“推进动作”，而不是面向“评分展示”。

#### Missing Context

明确提醒：

- 当前哪些判断因为缺会议转录、聊天记录、邮件上下文而不可靠
- 哪些建议需要先补信息再执行

### 10.5 一线销售不应看到什么

不应默认看到：

- 冗长对象表格
- 大量管理视角 KPI
- 跨团队经营指标

### 10.6 一线销售成功体验

一线销售打开页面后，应该在 1 分钟内完成：

1. 知道今天先做什么
2. 直接使用系统已准备好的草稿
3. 明白当前最大的 deal 阻塞是什么

## 11. 三个工作台之间的关系

三个工作台不是三套孤立页面，而是同一套经营系统的三个控制层。

关系应当是：

```text
CEO 工作台
看公司收入偏差和资源配置

↓

销售主管工作台
看团队风险、介入动作和 pipeline 真实性

↓

一线销售工作台
看客户推进动作、承诺和执行细节
```

同一件事情在三层的表达应不同：

- CEO 看到的是收入影响和是否要介入
- 销售主管看到的是谁该负责处理、如何管理
- 一线销售看到的是下一步具体推进动作

这正对应你们想要的分层：

- `Revenue Brain`
- `Pipeline Engine`
- `Deal Brain`

## 12. 页面交互原则

### 12.1 顶层永远先给结论

不要让用户先读图表再自己推断。

顺序必须是：

1. 结论
2. 原因
3. 建议
4. 证据入口

### 12.2 决策卡片必须支持完整闭环

每张关键卡片都应支持：

- 查看证据
- 追问为什么
- 修改建议
- 批准
- 驳回
- 延后

### 12.3 下钻逻辑必须明确

从工作台下钻应该分别进入：

- `Deal` 详情页查看具体推进证据
- `Meeting` 详情页查看原始理解依据
- `Pipeline` 页查看群体风险分布
- `Revenue` 页查看收入组成与情景变化

### 12.4 自主性必须有边界

顶层必须明确区分：

- 系统已完成判断
- 系统建议执行
- 系统等待批准
- 人必须亲自决定

### 12.5 全局 Agent 面板必须升级为对话工作区

全局 Agent 面板不能停留在“展示默认 prompt + 点击按钮询问”的阶段。

新的面板职责必须包括：

1. `收起 / 展开`
2. 自由输入问题
3. 自动携带当前页面 context
4. 支持把决策卡拖入对话框作为附加上下文
5. 围绕单次任务持续追问，而不是每次孤立问答

因此面板在交互上应更接近：

- Cursor 式的上下文对话工作区

而不是：

- 只会展示固定推荐语的侧边说明栏

### 12.6 决策卡与 Agent 对话必须连通

角色工作台里的 `Decision Card` 不应只是静态信息卡。

每张卡都应该天然支持进入 Agent 深聊，至少通过以下方式之一：

1. 点击围绕该卡继续追问
2. 拖拽到 Agent 对话输入区

推荐优先实现：

- 卡片可拖拽到 Agent 输入框
- 输入框显示已附加卡片
- 模型请求自动拼接卡片信息与当前页面上下文

这样用户不是在“跳页面”，而是在对当前决策对象继续协作。

## 13. 视觉与信息密度原则

为了避免继续长成 CRM，三套工作台必须遵守：

1. 减少大表格和密集列
2. 减少“总览式”平铺卡片
3. 强化主次层级，只突出少数最重要事项
4. 让卡片像“待决事项”和“作战简报”，而不是数据摘要块
5. 允许一页只聚焦 3 到 5 个最重要事项

建议视觉气质：

- 更接近 operating room / command center
- 不接近 CRM record system
- 不接近 BI dashboard wall

## 14. 与现有页面体系的关系

本轮不重做全站 IA。

建议新增三个一级页面：

- `/ceo-command-center` -> CEO 主控室
- `/sales-manager-cockpit` -> 销售主管驾驶舱
- `/sales-war-room` -> 一线销售作战室

现有 `/home` 保持不动，作为旧入口和通用首页继续保留。

现有页面暂时保留原职责：

- `/deals` 与 `/deals/[dealId]` 作为对象证据与执行页
- `/meetings` 与 `/meetings/[meetingId]` 作为会话理解与证据页
- `/pipeline` 作为群体推进和提议页
- `/revenue` 作为收入组成和风险下钻页

换句话说：

```text
先新增“角色决策入口层”，
后续再决定是否重构旧入口和对象层。
```

## 15. 后续实现建议

如果设计获批，下一步实现建议按以下顺序进行：

1. 先定义三套工作台共享卡片模型
2. 再重构 `/home` 三个角色视图
3. 保持 Deal / Meeting / Pipeline / Revenue 先不大改
4. 最后再决定是否把 Agent Workspace 也改成决策收件箱

---

## 16. 本轮设计验收标准

如果后续实现符合以下条件，可以认为三套工作台方向正确：

1. 打开任一工作台，用户先看到的是“重要决策”，不是对象列表
2. CEO 首页不再像 Revenue dashboard
3. 销售主管首页不再像 Pipeline dashboard
4. 一线销售首页不再像个人 CRM 工作区
5. 顶层每个重要判断都能下钻到证据
6. 顶层每个关键动作都能区分建议、批准、应用、同步
7. 顶层明确体现系统自主性，但不隐藏人工确认边界

---

## 17. 结论

本轮设计的核心不是重新排版首页，而是重新定义首页是什么。

新的定义应是：

```text
CEO、销售主管、一线销售分别拥有自己的 Agent 决策工作台。
CRM、会议、Deal、Pipeline 只是底层事实层和证据层。
顶层产品要呈现的是“判断、决策、介入、执行”，
而不是“字段、列表、图表、全量记录”。
```

推荐进入下一阶段的唯一方案：

```text
以“决策卡片优先”的角色工作台替代“传统 dashboard 式角色首页”。
```
