# AI Revenue Management OS 前端设计产品方案

## 1. 产品定位

### 1.1 产品名称
AI Revenue Management OS（销售智能操作系统）

### 1.2 产品使命
通过整合客户会议、邮件、聊天记录、电话与 CRM 数据，帮助 CEO、销售主管、一线销售在同一套系统中完成：

- 理解客户真实意图
- 判断每个 Deal 的真实推进状态
- 管理整体 Pipeline 健康度
- 预测收入
- 评估销售水平并支持培训、管理与激励

### 1.3 前端设计目标
前端不是“展示数据的后台”，也不只是“多图表的看板系统”，而是一个以 Agent 为核心交互入口的决策工作台。

目标有五个：

1. 让一线销售在单个客户场景下高效行动
2. 让销售主管在团队与 Pipeline 视角下快速判断风险
3. 让 CEO 在收入与关键商机视角下快速决策
4. 让系统天然承载多 Agent 协同产出的结果
5. 让用户感觉自己面对的不是冷冰冰的分析面板，而是一个“能理解上下文、能主动反馈、能对话协作”的销售 Agent

### 1.4 Agent 视角设计原则
本产品整体设计应以 Agent 视角为主，而不是纯看板视角。

这里的区别是：

- 看板视角的核心是“信息聚合与状态展示”
- Agent 视角的核心是“系统像一个可协作的人，理解上下文、主动反馈、可解释、可对话、可纠错”

因此本产品的前端设计原则不是“所有内容都做成聊天”，而是：

1. 重要结论由 Agent 主动说出来，而不是等用户自己读图表
2. 用户可以围绕任何结论继续追问“为什么”“依据是什么”“下一步怎么做”
3. 所有关键结论都要能追溯到证据
4. 所有关键建议都要允许人工确认、修改、驳回、重跑
5. Dashboard、列表、图表仍然保留，但它们属于 Agent 的工作界面，而不是产品的唯一主角

### 1.5 一句话定义
AI Revenue Management OS 前端不是一个 CRM 页面，而是一个让不同角色基于同一套客户互动数据，与销售 Agent 协作完成判断、推进与决策的智能工作台。

可以概括为：

**用客户互动重建 Deal，用 Deal 重建 Pipeline，用 Pipeline 重建 Revenue。**

## 2. 产品信息架构

整个前端按五层组织：

1. 数据接入层
2. 统一对象层
3. Intelligence 层
4. Agent 协作层
5. 角色工作台层

可视化理解如下：

```text
数据接入
↓
统一对象层（客户 / 联系人 / 会议 / 沟通 / Deal）
↓
Deal Intelligence
↓
Pipeline Intelligence / Revenue Intelligence
↓
Agent 协作层（解释 / 建议 / 纠错 / 对话）
↓
角色化工作台
```

### 2.1 统一对象模型
为了避免做成三套割裂页面，前端必须围绕统一对象模型展开。

核心对象建议定义为：

- Account（客户公司）
- Contact（联系人 / 决策人）
- Meeting（会议）
- Conversation（邮件 / 聊天 / 通话摘要）
- Deal（商机）
- Agent Output（Agent 输出结果）
- Forecast Snapshot（预测快照）
- Coaching Review（复盘与辅导结果）

不同角色看到的是同一套对象的不同视图与权限，而不是不同的数据口径。

### 2.2 Agent 协作层定义
Agent 协作层是本产品与传统 BI/CRM 前端的核心差异，至少承担四类能力：

- 主动反馈：告诉用户当前最重要的风险、变化和建议
- 可解释：告诉用户为什么得出这个判断
- 可对话：允许用户追问、改写、确认任务
- 可纠错：允许用户驳回、修正、重新生成 Agent 输出

## 3. 用户角色与视图策略

### 3.1 CEO / 创始人
关注：收入、风险、关键客户、季度预测

看到的核心内容：

- Revenue Forecast
- Pipeline Coverage
- Top Deals
- 高风险 Deal
- 团队整体趋势
- Agent 的关键经营提醒

颗粒度：公司级 / 大区级 / 团队级

Agent 交互重点：

- 用一句话总结本周收入风险
- 解释预测变化的原因
- 支持追问“哪些 Deal 影响最大”

### 3.2 销售主管 / VP / Head of Sales
关注：团队推进效率、Pipeline 真实性、销售水平、停滞商机

看到的核心内容：

- Pipeline Dashboard
- Deal Health 分布
- 阶段转化率
- 停滞 Deal
- 销售能力评分
- 会议亮点 / 不足复盘
- Agent 给出的辅导建议

颗粒度：团队级 / 销售个人级 / Deal 级

Agent 交互重点：

- 主动指出“本周最该处理的风险”
- 解释哪些 Deal 停滞、为什么停滞
- 协助生成辅导动作与复盘任务

### 3.3 一线销售
关注：我接下来该做什么，客户现在是什么状态，如何更高概率推进成交

看到的核心内容：

- 今日待办
- 会议准备 Brief
- 会议总结
- 客户异议
- Next Step 建议
- 自动跟进内容
- Agent 对当前 Deal 的推进建议

颗粒度：客户级 / Deal 级 / 单次会议级

Agent 交互重点：

- 给出明确下一步动作
- 支持直接对话生成跟进邮件和会议提纲
- 支持用户说“这条建议不对，重新来”

## 4. 前端总体结构

### 4.1 全局导航
建议采用左侧主导航 + 顶部上下文栏 + 全局 Agent 入口。

左侧主导航：

- 首页
- Agent Workspace
- 收入中心
- Pipeline
- Deals
- 客户
- 会议
- 销售团队
- 复盘与培训
- 数据接入
- 设置

顶部上下文栏：

- 搜索框（客户 / Deal / 邮件 / 会议）
- 时间筛选
- 团队筛选
- 销售筛选
- 快捷创建动作
- 通知中心
- Agent 状态入口
- 数据状态入口

### 4.2 页面层级
L1：角色首页 / Agent 首页  
L2：模块页（Pipeline、Deals、会议、销售团队）  
L3：详情页（单个 Deal、客户、会议、销售个人）  
L4：抽屉 / Side Panel（Agent 输出、记录、建议、操作、对话）  

### 4.3 Agent 作为全局交互层
不论用户在哪个页面，都应可以唤起 Agent。

全局 Agent 能力包括：

- 基于当前页面上下文发起提问
- 让 Agent 总结当前页面关键信息
- 围绕任一指标追问依据
- 让 Agent 生成后续动作
- 对 Agent 输出做确认、修改、驳回、重跑

### 4.4 数据状态的全局可见性
由于系统依赖多数据源，所有核心页面都必须让用户知道数据是否完整、是否新鲜。

建议全局显示三类状态：

- 数据覆盖率
- 最近同步时间
- 关键缺失告警

例如：

- 会议数据完整
- CRM 已同步 2 小时前
- 邮件数据缺失，风险判断可信度下降

## 5. 首页设计（角色化 Dashboard + Agent 首页）

### 5.1 首页的核心定位
首页不应只是静态 Dashboard，而应是“角色首页 + Agent 简报页”。

每个首页应回答四个问题：

1. 现在最重要的事情是什么
2. 为什么这件事重要
3. 我应该做什么
4. 我能不能继续追问或直接执行

### 5.2 CEO 首页
模块布局建议：

第一屏：

- 本季度预测收入
- 本季度目标达成率
- Pipeline Coverage
- 高风险金额
- Agent 经营摘要

第二屏：

- Top 10 Deals
- 阶段分布漏斗
- 大客户风险地图
- 收入趋势图

第三屏：

- 组织层警报
- 区域 / 团队对比
- Agent 发现摘要

核心卡片：

- Revenue Forecast Card
- Pipeline Risk Card
- Forecast Accuracy Card
- Strategic Deal Watchlist
- CEO Brief Card

### 5.3 销售主管首页
第一屏：

- 本周 Pipeline 健康度
- 本周新增商机
- 停滞 Deal 数量
- 团队销售能力评分
- Agent 风险摘要

第二屏：

- Deal 风险列表
- 销售推进速度排行
- 阶段转化率漏斗
- 销售行为洞察

第三屏：

- 团队辅导建议
- 会议复盘待看
- 高价值会议回放

### 5.4 一线销售首页
第一屏：

- 今日待办
- 待跟进客户
- 本周会议
- 我的 Pipeline 健康度
- Agent 今日建议

第二屏：

- 推荐 Next Step
- 新产生的 Agent 建议
- 本周高风险客户
- 会议准备入口

第三屏：

- 会议总结历史
- 自动生成邮件草稿
- 个人成长建议

### 5.5 首页上的 Agent 表达方式
首页中的 Agent 不应只是一个角落按钮，而应以“会说话的摘要卡”出现。

建议形式：

- 一句结论
- 两到三条原因
- 一个明确下一步
- 两个快捷追问

例如：

- “本周团队 Pipeline 总体健康，但有 3 个后期 Deal 风险显著升高。”
- “原因：两场关键会议缺少明确预算承诺；一个高金额 Deal 14 天无推进；一位销售连续两次未锁定决策人。”

## 6. AI 结论的解释标准

### 6.1 为什么要单独定义
本系统的核心判断依赖 AI 与多源数据，因此所有关键结论都必须遵守统一的解释标准，否则用户会把系统视为黑盒。

### 6.2 适用范围
以下结论必须可解释：

- Deal Health
- Win Probability
- Forecast Confidence
- Pipeline Risk
- Coaching Score
- Next Step 建议
- 会议亮点 / 不足

### 6.3 每个 AI 卡片的最小信息结构
每个 AI 结论卡片至少包括：

1. 结论
2. 置信度
3. 最近更新时间
4. 数据时间范围
5. 数据覆盖率
6. 三条以内关键依据
7. 查看原始证据入口
8. 纠错 / 重跑入口

### 6.4 证据来源要求
可作为证据的来源包括：

- 会议转录片段
- 邮件摘要
- 聊天记录摘要
- CRM 字段变化
- 历史推进时间线
- 同类 Deal 统计模式

### 6.5 数据缺失时的降级原则
当关键数据缺失时，页面不能继续“假装确定”。

应该明确展示：

- 数据不完整
- 可信度下降
- 缺了哪些数据
- 哪些判断暂不建议采信

## 7. Deals 模块设计

### 7.1 Deal 列表页
目标：快速识别哪些 Deal 值得关注，并让用户直接进入 Agent 协作。

字段建议：

- Deal 名称
- 客户名称
- 所属销售
- 当前 Stage
- Deal Health
- Win Probability
- 金额
- 下一步行动
- 下次会议时间
- 风险等级
- 数据状态
- Agent 最新提醒

交互：

- 可按 Stage / 健康度 / 金额 / 销售 / 时间排序
- 支持保存视图
- 支持风险筛选
- 支持批量进入 Review 模式
- 支持一键唤起 Agent 解释“为什么它是高风险”

### 7.2 Deal 详情页
建议采用“详情页 + Agent 侧边协作区”的结构，而不是单纯四栏堆信息。

#### A. 左侧：概览区

- Deal 标题
- 客户信息
- 金额
- Stage
- 所属销售
- 更新时间
- 数据完整度

#### B. 中间主内容区

- Deal Health 卡片
- Win Probability 卡片
- 风险信号
- 客户需求摘要
- 决策链路
- 历史推进时间线
- 数据状态提示

#### C. 右侧：Agent 协作区

- Next Step 建议
- 生成邮件
- 生成会议 Brief
- 请求经理 Review
- 推送到 CRM
- 追问 Agent
- 驳回 / 修正 / 重跑建议

#### D. 底部标签页

- 会议记录
- 邮件与聊天
- 活动日志
- Agent 分析
- 文件（第二阶段再完整补齐）

### 7.3 Deal 核心前端组件

- Health Score Ring
- Stage Progress Tracker
- Risk Signal Chips
- Decision Maker Map
- Buying Signal Timeline
- Next Step Action Box
- Evidence Drawer
- Data Freshness Badge
- Agent Chat Panel

### 7.4 Deal 页面的关键语义定义
`Deal` 页面中所有关键动作必须语义明确。

尤其需要明确以下区别：

- 建议：Agent 给出的判断，不自动落库
- 确认：人工接受该建议
- 应用：将确认后的内容写入系统字段
- 同步：将系统字段回写 CRM

不能让用户误以为“看到一个建议”就等于“系统已经修改真实状态”。

## 8. Pipeline 模块设计

### 8.1 Pipeline Dashboard
目标：帮助销售主管理解整体推进状态，并能从风险分布快速进入具体 Agent 对话。

核心区域：

- 阶段漏斗
- 每阶段金额与数量
- 阶段转化率
- 平均停留时长
- 停滞 Deal 分布
- 风险 Deal 热力图
- Agent 风险摘要

### 8.2 视图模式
建议支持三种切换：

- 漏斗视图
- 看板视图
- 健康度矩阵视图

#### 漏斗视图
适合看阶段转化与整体量级。

#### 看板视图
按 Stage 展示 Deal 卡片，但这里的拖拽默认定义为“模拟调整”，不是直接修改真实 Stage。

必须补充以下交互语义：

- 拖拽仅用于模拟或形成 review 草案
- 模拟结果需明确标记“尚未应用”
- 若用户希望正式变更，需二次确认
- 正式变更后需区分“仅系统内更新”还是“同步回 CRM”
- 需要权限控制、审计日志和变更原因

#### 健康度矩阵
X 轴：Stage  
Y 轴：Deal Health  
用于识别“阶段靠后但健康度低”的危险商机。

### 8.3 关键图表

- Stage Funnel
- Conversion Trend
- Aging Distribution
- Risk Matrix
- Rep Comparison Chart

### 8.4 Pipeline 页面上的 Agent 角色
Pipeline 页面中的 Agent 应承担“解释异常”和“生成行动”的责任。

例如：

- 为什么本周转化率下降
- 哪些 Deal 导致风险暴露上升
- 如果延迟某几个 Deal，会如何影响季度预测
- 应该先让哪位经理介入哪个 Deal

## 9. 收入中心设计（Revenue Center）

### 9.1 核心目标
给 CEO / VP 一个可操作的收入视图，而不是静态报表。

### 9.2 页面结构

- Forecast 总览
- 收入组成结构
- 高价值 Deal 追踪
- 预测偏差分析
- 风险场景模拟
- Agent 收入简报

### 9.3 核心卡片

- Forecast Total
- Commit / Best Case / Upside
- Forecast Confidence
- Risk Exposure
- Top Revenue Drivers

### 9.4 高价值交互

- 可点进任一收入数字查看组成 Deal
- 支持按地区 / 团队 / 产品线切换
- 支持“如果某个 Deal 延迟”的场景模拟
- 支持让 Agent 解释预测变化

### 9.5 可解释性要求
收入中心中的每个关键数字，都必须允许用户看到：

- 由哪些 Deal 构成
- 最近变动来自哪里
- 置信度为何升高或降低
- 哪些数据缺失影响预测质量

## 10. 会议模块设计

### 10.1 会议列表页
展示所有客户会议及状态：

- 待准备
- 已完成待复盘
- 已生成总结
- 已同步到 Deal
- 需人工确认

字段：

- 会议主题
- 客户
- 销售
- 时间
- 会议类型
- 是否生成总结
- 是否有风险信号
- 数据状态

### 10.2 会议详情页
建议拆成三段式：

#### 会前

- 客户背景
- 历史沟通总结
- 会议目标
- 推荐提问清单
- Agent 会议准备建议

#### 会中（事后展示）

- 转录文本
- 关键片段标记
- 客户信号
- 销售行为标注

#### 会后

- 会议总结
- 亮点
- 不足
- Action Items
- 关联 Deal 更新
- 人工确认入口

### 10.3 会议详情页核心组件

- Transcript + Highlights
- Signal Tagging Panel
- Meeting Summary Card
- Coaching Review Card
- Action Items List
- Related Deal Snapshot
- Evidence Drawer
- Agent QA Panel

### 10.4 会议总结的纠错闭环
会议总结不能只有“生成”没有“纠错”。

必须支持：

- 标记摘要错误
- 手动修正重点结论
- 重新生成总结
- 查看引用片段
- 确认哪些内容已同步到 Deal

## 11. 销售团队模块设计

### 11.1 团队总览

- 团队业绩
- 团队 Pipeline
- 团队健康度
- 团队平均销售能力评分
- 本周需要辅导的销售
- Agent 的团队辅导建议

### 11.2 销售个人页
核心目标：帮助主管判断谁该奖、谁该带、谁该训练。

内容：

- 个人 Pipeline
- 个人成交率
- 平均 Deal Health
- 销售能力评分
- 会议亮点与不足统计
- 建议培训主题
- Agent 个人辅导摘要

### 11.3 销售能力评估维度

- 提问能力
- 倾听能力
- 痛点挖掘能力
- 异议处理能力
- 推进能力
- 商务推进能力
- 会议掌控能力

展示形式：

- 雷达图
- 评分趋势图
- 同团队基准对比

### 11.4 能力评分的解释要求
能力评分不能只有分数，必须解释来源。

需要展示：

- 评分变化趋势
- 主要依据会议或片段
- 相比团队基准的差异
- 可执行的训练建议

## 13. Agent Workspace 设计

### 13.1 模块定位
`Agent Workspace` 是整个产品的核心交互层，用于让用户感觉系统是一个可协作的销售 Agent，而不是一堆分析卡片。

### 13.2 核心能力

- 基于上下文主动给出提醒
- 围绕 Deal / Meeting / Pipeline / Revenue 进行自然语言对话
- 对任一结论追问“为什么”
- 让 Agent 帮用户生成执行动作
- 对 Agent 的输出进行确认、修改、驳回、重跑

### 13.3 推荐 UI 结构

#### A. 对话主区

- Agent 对话流
- 上下文切换提示
- 快捷追问
- 建议动作卡片

#### B. 当前上下文区

- 当前 Deal / Meeting / Team / Forecast 对象
- 当前时间范围
- 当前引用的数据源
- 当前缺失的数据提醒

#### C. 证据区

- 会议片段
- 邮件摘要
- CRM 变化记录
- 时间线依据

#### D. 行动区

- 创建任务
- 生成邮件
- 生成 Brief
- 请求 Review
- 应用到系统
- 同步到 CRM

### 13.4 Agent 应具备的人感反馈
用户提出的“Agent 的核心是感觉反馈像人，以及可以跟她对话”是成立的，这应成为产品设计要求。

具体可以落成以下体验：

- Agent 先给一句自然语言判断，而不是先给图表
- Agent 会根据角色改变表达方式
- Agent 会记住当前上下文，不需要用户重复说明
- Agent 能接受模糊指令，如“这个 Deal 怎么推”“帮我给老板一句总结”
- Agent 的语气专业、简洁、像一个懂销售业务的同事

### 13.5 Agent 的可信度建设
为了避免“像人但不可信”，Agent 必须同时满足：

- 像人一样表达
- 像系统一样可追溯

因此所有对话型输出都要保留：

- 结论依据
- 数据来源
- 最近更新时间
- 人工确认状态
- 是否已应用 / 是否已同步

### 13.6 Agent 的纠错闭环
必须支持以下动作：

- 这条结论不对
- 依据不足
- 重新生成
- 改成这个版本
- 只保存为草稿
- 确认并应用

- 可读取字段说明

- 可回写字段说明

### 14.3 关键设计点

- 强调隐私与权限说明

- 清晰显示哪些字段会被读取、分析、回写
- 清晰展示数据缺失将影响哪些 AI 判断

### 14.4 数据状态如何透传到业务页面
数据接入状态不能只停留在接入模块页。

在 `Deal`、`Pipeline`、`Revenue`、`Meeting` 页面都应显示轻量状态提示，例如：

- 会议数据完整
- 邮件同步中断
- CRM 最近 24 小时未更新
- 日历权限未授权，会议建议可能不完整

## 15. 核心页面流程

### 15.1 一线销售的日常链路
首页 → 今日待办 → 某场会议 Brief → 与 Agent 讨论会议目标 → 会议结束 → 查看总结 → 修正要点 → 接收 Next Step → 生成跟进邮件 → 确认并发送

### 15.2 销售主管的日常链路
首页 → 查看 Agent 风险摘要 → 进入 Pipeline Dashboard → 点开高风险 Deal → 查看关联会议 → 追问风险原因 → 查看销售复盘 → 指派辅导动作

### 15.3 CEO 的日常链路
首页 → Revenue Forecast → 查看 Agent 的经营摘要 → 追问风险来源 → 查看 Top Deals → 要求主管跟进

### 15.4 围绕 Agent 的通用链路
任意页面 → 唤起 Agent → 让 Agent 解释结论 → 查看证据 → 修改建议 → 确认动作 → 应用到系统 / 同步到 CRM

## 16. 设计系统建议

### 16.1 设计原则

- 信息密度高，但层次必须清晰
- 先给结论，再给证据
- 风险信息优先高亮
- 角色视图隔离，避免信息噪音
- 所有 AI 结论必须可解释
- 所有 Agent 结果必须可确认、可修改、可驳回
- 对话体验要像人，但系统行为要可审计

### 16.2 组件体系
建议建立以下基础组件：

- 分数环
- 风险标签
- Stage 标签
- Agent 输出卡片
- Agent 对话卡片
- 证据抽屉
- 复盘卡片
- 漏斗图
- 时间线
- 角色切换器
- 对话摘录组件
- 行动建议组件
- 数据新鲜度徽标
- 可信度标签

### 16.3 色彩策略

- 绿色：健康 / 可推进
- 黄色：关注 / 中风险
- 红色：高风险 / 停滞
- 蓝色：AI 建议 / 信息类
- 紫色：收入 / 预测类
- 灰色：数据缺失 / 待确认 / 未同步

## 17. MVP 前端范围建议

### 17.1 MVP 原则
第一版不能把“页面数量少”误当成“MVP 足够小”。

MVP 应只围绕以下四件事：

- 会议理解
- Deal Health
- Next Step
- Pipeline 风险

### 17.2 MVP 必做页面

1. 登录后的销售主管首页
2. Deal 列表页
3. Deal 详情页
4. 会议详情页
5. Pipeline Dashboard
6. 全局 Agent 侧边协作区

### 17.3 MVP 必做能力

- Agent 风险摘要
- Deal Health 的可解释卡片
- Next Step 建议与人工确认
- 会议总结查看、修正、再生成
- Pipeline 风险识别
- 数据状态透传

### 17.4 MVP 可后置能力
以下内容不建议在首版做全量：

- Deal 页完整文件管理
- 全量邮件 / 聊天统一收件箱
- CRM 全自动双向同步策略配置
- 复杂收入模拟
- 完整销售能力评分体系
- 完整培训素材库

### 17.5 MVP 中最关键的交互闭环
首版必须走通这条链路：

会议完成 → Agent 生成总结 → 用户查看证据 → 修正重点 → Agent 生成 Next Step → 用户确认 → 应用到 Deal → 选择是否同步 CRM

## 19. 风险与前置定义

### 19.1 必须前置定义的产品语义
以下概念必须在设计和研发前统一：

- 建议 vs 确认 vs 应用 vs 同步
- 模拟调整 vs 正式变更
- Agent 输出草稿 vs 已确认结果
- 数据完整 vs 数据缺失
- 低置信度提醒 vs 高风险提醒

### 19.2 容易引发返工的地方

- 看板拖拽若没有语义定义，会导致 CRM 同步与权限返工
- AI 卡片若没有统一解释标准，会导致每页逻辑割裂
- 没有数据新鲜度展示，会让用户误信系统判断
- 没有纠错闭环，Agent 很难建立信任

---

## 20. 最终产品总结

这套前端最终不应被设计成单纯的看板系统。

它应该是：

- 以 Dashboard 承载全局判断
- 以对象页承载证据与操作
- 以 Agent 承载对话、解释、建议与纠错

如果只做看板，产品会像一个“更智能的 CRM”。
如果按 Agent 视角设计，产品才会更接近“一个懂销售、懂上下文、会主动反馈、能协作推进的智能同事”。
