# Conversational Agent OS Design

日期：2026-03-12

## 1. 目标

开发一个新版 `会话版 Agent OS`，替代当前“左侧菜单 + 中间卡片 + 右侧 Agent 面板”的重型结构。

新版本的核心假设：

- Agent 被当作“人”来协作，而不是页面里的辅助面板
- 用户主要通过对话与 Agent 协作
- 决策卡片不再是独立页面块，而是会话中的一种消息
- Agent 与 Agent 之间的 handoff 也以消息流的形式被前端可视化

## 2. 产品立场

不是把整个产品做成聊天工具。

而是：

- 会话承担协作
- 卡片承担决策
- 详情承担下钻

一句话：

`Conversation-first shell, decision-card payloads, explainable drill-down.`

## 3. 为什么要重做

当前版本的问题：

- 页面分区太强，用户在“看系统”而不是“和 Agent 协作”
- 中间卡片和右侧 Agent 面板分裂成两套交互
- 组织内消息流、上报、转发、批准、打回没有统一的会话主线
- 视觉和交互都偏重，像一个叠加了 Agent 的 CRM

新版本要解决的是：

- 让用户自然地把 Agent 当成业务伙伴
- 让决策、确认、上报、转发都发生在同一条线程里
- 让 Agent-to-Agent 通信对人可见，但不泄漏不该可见的内部内容

## 4. 核心交互模型

### 4.1 主界面结构

新版 IA 改成三层：

1. 左侧：线程与 Agent 列表
2. 中间：会话主舞台
3. 详情层：卡片详情 / 证据 / Deal / Meeting 下钻

不再保留固定右侧 Agent 面板。

### 4.2 左侧不再是“页面菜单”

左侧应切换为：

- `CEO Agent`
- `销售主管 Agent`
- `一线销售 Agent`
- 可选：`系统流转` / `最近 handoff`

更进一步，左侧每一项本质上是一个 `Thread`，而不是一个功能页面。

## 5. 会话主舞台

中间主舞台是消息流。

消息流里混排四种内容：

1. 自然语言消息
2. 决策卡消息
3. 系统流转消息
4. Agent handoff 消息

例如：

- “杨文星已将这次会议判断提交给销售主管 Agent”
- “销售主管 Agent 已将报价边界问题升级给 CEO Agent”
- “CEO Agent 已返回决策，生成一张新的主管执行卡”

## 6. 卡片在新系统里的角色

卡片不是附件，而是消息的一种。

### 6.1 卡片的一级展示

在消息流里，一张卡只展示：

- 标题
- 一句优先级判断
- 一行来源 / 信任信息
- 一个主动作按钮
- 一个 `查看详情`

### 6.2 卡片的二级展示

点开后才进入详情层，查看：

- 完整正文
- 流转轨迹
- 证据
- 相关 Deal / Meeting / 客户线程

### 6.3 卡片的即时操作

摘要层可以直接：

- `确认`
- `上报`
- `批准`
- `打回`

这样比“先点进去再找动作”更自然。

## 7. 发送内容类型

用户在会话里能发的内容只有两类：

1. 自然语言
2. 卡片

卡片发送后，本质上是一次正式 handoff。

例如：

- 销售把一张会议后生成的推进卡发给销售主管 Agent
- 销售主管把一张升级卡发给 CEO Agent

## 8. Agent-to-Agent 通信

### 8.1 表层体验

看起来像群聊或多方协作空间。

### 8.2 底层模型

底层不能是真正的完全对等群聊。

必须是：

- 多线程
- 带权限
- 带可见性范围
- 可追踪 handoff

因为：

- 下游不应该看到上游全部内部判断
- CEO Agent 的完整处理上下文不该默认暴露给一线销售
- 用户看到的应是“与自己相关的结果”，不是全部系统内部消息

## 9. 可见性模型

每条消息 / 每张卡片都要有 `visibility_scope`。

至少支持：

- `private_to_actor`
- `visible_to_current_level`
- `visible_to_upstream`
- `visible_to_downstream`
- `system_only`

这能支持：

- Agent 之间完整通信存在
- 人只看到自己该看到的结果
- 同时保留“系统真的在协作”的可解释性

## 10. @ 机制

支持 `@`，但第一版不做完全自由路由。

MVP 只支持受约束的升级关系：

- 一线销售 -> 销售主管 Agent
- 销售主管 -> CEO Agent
- 一线销售直接 @ CEO Agent：默认不开放，或只允许“申请升级”

这样既像真实组织，也避免线程失控。

## 11. Pinned Priority

虽然主界面变成会话流，但不能丢失优先级。

所以会话顶部必须始终固定一个区域：

- `当前最重要的一张卡`
- `为什么是它`
- `当前主动作`

否则系统会退化成聊天工具。

## 12. 对象模型

新版至少要把这 5 个对象做成一等公民：

### `Thread`

谁和谁在协作的会话上下文。

### `Message`

自然语言、系统消息、handoff 消息、卡片消息的统一容器。

### `DecisionCard`

可确认、可转发、可升级、可打回的决策对象。

### `Handoff`

一个 Agent 将某件事正式交给另一个 Agent 的记录。

### `VisibilityRule`

规定某条消息和某张卡对谁可见。

## 13. MVP 建议

不要一上来做复杂群聊。

第一版只做 3 个可交互线程：

1. `CEO Agent`
2. `销售主管 Agent`
3. `一线销售 Agent`

每个线程里都能：

- 输入自然语言
- 发送卡片
- 接收卡片
- 直接确认摘要层动作
- 查看消息化的 handoff 记录

## 14. 第一版推荐流转

### 14.1 一线销售线程

- 一线销售和一线销售 Agent 协作
- 会议后生成总结与推进卡
- 一线销售确认或补充后，上报给销售主管 Agent

### 14.2 销售主管线程

- 销售主管和销售主管 Agent 协作
- 接收一线销售上报
- 决定派发、打回、升级

### 14.3 CEO 线程

- CEO 和 CEO Agent 协作
- 接收主管升级
- 决定批准、驳回、要求补充、亲自介入

## 15. 页面与路由建议

第一版不要替换旧页面。

建议：

- 保留现有 `/agent-task-cards`
- 新开一个独立入口，例如 `/conversational-agent-os`

这样可以：

- 保留旧版 demo 作为稳定基线
- 新版单独迭代，不被旧结构拖累
- 方便两种交互直接对比

## 16. 设计方向

视觉上不应像 IM 工具，也不应像 CRM。

建议方向：

- 左侧线程更像“作战对象列表”，不是通讯录
- 中间消息流更像“决策时间线”，不是闲聊流
- 卡片在流里要有明显权重，但不做大块堆叠
- 系统流转消息要克制，像运营日志，不抢主舞台

## 17. 风险

### 风险 1：做成普通聊天工具

修正：

- 顶部固定优先级卡
- 卡片始终是最强决策单元

### 风险 2：做成假群聊

修正：

- 底层用线程 + handoff + visibility
- 不做所有人完全平等可见

### 风险 3：交互过重

修正：

- MVP 不做复杂线程树
- 只做 3 个核心 Agent 线程

## 18. 下一步

如果这个方向成立，下一步不是直接写代码，而是继续补两份规格：

1. `会话 IA 与消息类型设计`
2. `MVP 实现计划`

确认后再进入新版独立页面开发。
