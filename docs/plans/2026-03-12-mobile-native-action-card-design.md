# Mobile Native Action Card Design

## Goal

把 `/design-system` 里的所有卡片，从“分析报告卡”收敛成“移动端原生行动卡”。

## Problem

当前卡片的问题不是单一视觉问题，而是结构问题：

- 卡面常驻信息过多，解释盖过动作
- 卡中有卡，层级超过移动端阅读负荷
- 标题、正文、来源、按钮同时抢注意力
- 渐变、阴影、重边框把卡做成了 Web3 面板，而不是任务卡

结果是用户第一眼不知道这是一张要行动、要判断，还是只要阅读的卡。

## Design Principle

### 1. Card As Action Surface

卡面只负责一件事：

> 让用户在一屏内知道“现在该做什么”

解释、来源、更新时间、证据，不再常驻卡面，统一进入详情层。

### 2. One Screen, One Action

所有卡片都使用同一套移动端骨架：

- `对象`
- `任务`
- `关键判断`
- `可信度 / 执行状态`
- `主动作`

其中第 4 行允许按卡片类型微调：

- 判断型卡：显示 `可信度`
- 命令型卡：显示 `执行状态`

### 3. Detail Is For Evidence, Not For Summary

点击卡片后进入详情页，详情页负责回答：

- 为什么这样判断
- 信息来源
- 最近更新
- 证据片段
- 数据完整度 / 可信度

卡面不再承担报告页职责。

## Card Mapping

### Fact Cards

- 对象：客户 / 客户线程
- 任务：本次变化要推动什么
- 关键判断：最重要的真实变化
- 可信度
- 主动作：查看详情 / 生成下一步

### Rep Decision Cards

- 对象：客户
- 任务：当前推进目标
- 关键判断：先做什么，不做什么
- 可信度
- 主动作：执行下一步

### Manager Decision Cards

- 对象：客户线程 / 下属
- 任务：要不要介入、调配、培训、上报
- 关键判断：主管为什么要出手
- 可信度
- 主动作：发起介入

### CEO Decision Cards

- 对象：关键客户 / 战略事项
- 任务：是否投入资源、是否亲自出面
- 关键判断：CEO 该不该成为资源的一部分
- 可信度
- 主动作：批准资源 / 安排行程

### Escalation Cards

- 对象：哪条线、谁发起
- 任务：希望上层拍什么板
- 关键判断：为什么必须上升
- 可信度
- 主动作：继续上报 / 退回补充

### Command Cards

- 对象：执行人
- 任务：必须完成什么
- 关键判断：执行重点是什么
- 执行状态
- 主动作：开始执行 / 回传结果

## Visual Direction

### Overall

- 页面外壳继续保留深色作战室
- 卡片本体改成更轻、更扁平、更像原生任务卡

### Avoid

- 大面积高饱和渐变
- 厚重发光阴影
- 卡中卡堆叠
- 500px 以上的超长卡

### Target

- 卡片高度优先控制在 `240px ~ 280px`
- `padding` 控制在 `16px ~ 18px`
- 主按钮高度 `44px`
- 标题不再做超大展示
- 正文只保留一句关键判断，不常驻解释段落

## Implementation Direction

- 为 `/design-system` 中的真实卡墙建立统一的行动卡数据骨架
- 为 `DesignVariantGallery` 和 `CardWall` 复用同一套“原生行动卡”阅读结构
- 详情层统一承接解释、来源、更新时间、证据
- 在设计系统页里保留已有 `Background Directions` 和 `Universal Card Container`，但示意内容同步更新为新骨架
