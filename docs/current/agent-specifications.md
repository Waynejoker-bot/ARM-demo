# Agent 规格定义

日期：2026-03-17
状态：canonical

> 架构图见 `docs/diagrams/agent-architecture-relationships.mmd`
> 会后实际流程图见 `docs/diagrams/post-meeting-flow.mmd`

---

## 总览

| Agent | 类型 | 实例化规则 | 一句话定义 |
|-------|------|-----------|-----------|
| 主管 BPAgent | BPAgent | 1 主管 = 1 实例 | 销售主管的业务伙伴，帮助管人、管事、管资源 |
| 一线销售 BPAgent | BPAgent | 1 销售 = 1 实例 | 一线销售的业务伙伴，帮助理解素材、推进商机、闭环执行 |
| 客户 Agent | 客户 Agent | 1 客户 = 1 实例 | 维护单个客户的完整生命周期和关系上下文 |
| 商机 Agent | 客户 Agent 子集 | 1 商机 = 1 实例，挂在所属客户 Agent 下 | 管理单个 Deal 从立项到成交的全流程 |

### 多对关系

```
主管 BPAgent ×1 ──辅导/审批──→ 销售 BPAgent ×N
销售 BPAgent ×1 ──操作──→ 客户 Agent ×M（一个销售负责多个客户）
客户 Agent ×1 ←──被操作──── 销售 BPAgent ×K（一个客户可被多个销售协作）
客户 Agent ×1 ──包含──→ 商机 Agent ×P（一个客户下可有多个商机）
```

## 1. 主管 BPAgent

**简介**：销售主管的 Business Partner。不替代主管做决定，而是帮主管看清全局、发现问题、高效干预。

### 触发时机

| 触发方式 | 触发条件 | 说明 |
|---------|---------|------|
| 定时主动 | 每日开盘 / 每周汇总 | 自动聚合团队数据，推送团队状态卡片 |
| 被动 | 收到销售 BPAgent 上报 | 销售求助、审批请求触发 |
| 被动 | 商机 Agent 风险升级 | 某商机风险持续升高，自动升级到主管 |
| 被动 | 主管主动发起对话 | 主管在对话界面提问或下指令 |

### 目标 1.1：团队状态监控

| 维度 | 说明 |
|------|------|
| **目标** | 让主管快速了解团队整体状态，识别需要关注的人和事 |
| **输入** | 各销售 BPAgent 的汇总数据（跟进活跃度、商机推进速率、任务完成率）；Pipeline 全局快照；逾期/停滞商机列表 |
| **输出** | 团队日报/周报卡片；异常预警卡片（谁掉链子了、哪个商机卡住了）；团队排名/趋势摘要 |
| **动作** | 主动推送团队状态卡片 → 主管可追问某个销售或商机的详情 → 下钻到具体销售 BPAgent 或客户 Agent |

### 目标 1.2：辅导与培训

| 维度 | 说明 |
|------|------|
| **目标** | 基于真实会议和跟进数据，帮主管针对性辅导销售 |
| **输入** | 具体销售的会议录音/转写；该销售的商机推进历史；客户互动记录（邮件、聊天） |
| **输出** | 辅导建议卡片（"销售A在客户X的会议中没有推进预算讨论，建议关注"）；话术参考；最佳实践对比（同团队其他销售在类似场景的做法） |
| **动作** | 主动推送辅导建议 → 主管审阅并编辑 → confirm → 发送给对应销售 BPAgent 作为辅导任务卡片 |

### 目标 1.3：审批与资源调配

| 维度 | 说明 |
|------|------|
| **目标** | 处理销售上报的审批请求和资源需求，快速做出决定 |
| **输入** | 销售 BPAgent 上报的审批请求（折扣审批、特殊条款、客户介入请求）；关联商机的当前状态和历史；资源占用情况 |
| **输出** | 审批决策卡片（批准/驳回/附条件批准）；资源调配指令（"安排售前工程师支持客户X的 Demo"）；主管介入卡片（主管亲自跟进某个客户/商机） |
| **动作** | 收到销售上报 → 推送审批卡片给主管（附背景和建议） → 主管决策 → confirm → apply（更新商机状态） → 通知对应销售 BPAgent |

## 2. 一线销售 BPAgent

**简介**：一线销售的 Business Partner。是销售每天打开的第一个工具，帮销售从"信息"到"行动"。

### 触发时机

| 触发方式 | 触发条件 | 说明 |
|---------|---------|------|
| 被动 | 销售发送素材 | 会议链接、录音、聊天截图、邮件等 → 触发素材理解 |
| 被动 | 销售发起对话/追问 | 销售在对话界面提问 → 触发对话响应 |
| 被动 | 销售 confirm 动作 | 销售确认建议 → 触发 apply 到客户/商机 Agent |
| 被动 | 收到客户/商机 Agent 信号 | 风险变化、阶段推进建议等 → 聚合后推送给销售 |
| 被动 | 收到主管 BPAgent 辅导/指令 | 主管下发的辅导任务或审批结果 → 推送给销售 |
| 定时主动 | 每日开盘 | 聚合所有商机/客户状态 → 推送今日关注 Top 3 |
| 定时主动 | 会议前 30 分钟 | 检测日历 → 推送会议准备卡片 |

### 目标 2.1：素材理解与结构化

| 维度 | 说明 |
|------|------|
| **目标** | 把销售上传的原始素材转化为可操作的结构化信息 |
| **输入** | 会议录音/转写（核心）；聊天截图；邮件；文档/链接；手动输入的文字备注 |
| **输出** | 会议摘要卡片（关键议题、决策、下一步、参会人态度）；客户情报卡片（从聊天/邮件中提取的关键信号）；商机更新建议（阶段变化、新增风险、关键人变化） |
| **动作** | 销售上传素材 → Agent 分类识别 → 生成结构化卡片 → 销售可追问细节（"会上他们到底怎么说预算的？"） → 销售 confirm 摘要 → apply 到对应客户/商机 Agent |

### 目标 2.2：日常任务推送

| 维度 | 说明 |
|------|------|
| **目标** | 每天主动告诉销售"今天最该做什么"，减少遗漏和低效 |
| **输入** | 销售名下所有商机状态（来自客户 Agent / 商机 Agent）；日历/会议安排；逾期任务；客户最近信号（回复了邮件、没回复、打开了方案文档等） |
| **输出** | 今日关注卡片（Top 3 待办，含优先级和原因）；跟进提醒卡片（"客户Y 3天没回复，建议换个角度跟进"）；会议准备卡片（"下午2点和客户X开会，以下是上次会议遗留问题"） |
| **动作** | 每日开盘主动推送 → 销售查看 → 可追问（"为什么这个排第一？"） → 销售接受或调整优先级 → 标记开始执行 |

### 目标 2.3：执行闭环

| 维度 | 说明 |
|------|------|
| **目标** | 帮销售把决定变成真实动作，确保不停留在"想做"阶段 |
| **输入** | 销售确认的下一步动作（confirm 后的任务）；商机当前阶段和条件；CRM 字段现状 |
| **输出** | 执行确认卡片（"已将商机X从'需求确认'推进到'方案评估'"）；同步状态卡片（"已/未同步到CRM，原因：…"）；后续跟进建议（"阶段推进后，建议下一步做…"） |
| **动作** | 销售 confirm 动作 → apply 到内部状态（更新商机 Agent / 客户 Agent） → 可选 sync 到外部 CRM → 推送执行结果卡片 → 进入下一轮循环 |

### 目标 2.4：向上求助与上报

| 维度 | 说明 |
|------|------|
| **目标** | 当销售遇到超出权限或能力范围的事，帮助快速上报给主管 |
| **输入** | 销售发起的上报请求（"需要主管帮忙""这个折扣我批不了"）；关联商机/客户的完整上下文 |
| **输出** | 上报卡片（发送给主管 BPAgent，含完整上下文和销售的诉求）；等待反馈卡片（告知销售"已上报，主管还未回复"） |
| **动作** | 销售发起求助 → Agent 打包上下文 → 推送给主管 BPAgent → 主管处理后 → 结果回传给销售 BPAgent → 推送反馈卡片给销售 |

## 3. 客户 Agent

**简介**：一个客户的"数字记忆体"。不面向任何人类用户直接交互，而是为 BPAgent 提供关于这个客户的所有结构化事实和判断。

### 触发时机

| 触发方式 | 触发条件 | 说明 |
|---------|---------|------|
| 被动 | 收到销售 BPAgent apply 的新素材 | 会议摘要确认后写入 → 触发 Account Thread 更新和健康度重算 |
| 被动 | 商机 Agent 回写状态 | 商机阶段变更、赢/丢单等 → 更新 Account Thread |
| 定时主动 | 定期数据新鲜度检查 | 标注过期数据、提醒补充缺失信息 |
| 被动 | BPAgent 查询客户上下文 | 销售追问客户情况时，提供结构化响应 |

### 目标 3.1：客户关系维护与健康度评估

| 维度 | 说明 |
|------|------|
| **目标** | 持续跟踪客户关系状态，识别关系升温或降温信号 |
| **输入** | 该客户相关的所有会议记录（由销售 BPAgent apply 过来）；邮件/聊天互动频率和内容摘要；关键联系人变化；CRM 历史字段 |
| **输出** | 客户健康度评分（含置信度和计算依据）；关系趋势（上升/平稳/下降 + 原因）；关键联系人图谱（谁是决策人、影响者、使用者） |
| **动作** | 接收新素材 → 更新 Account Thread → 重新计算健康度 → 如果有显著变化 → 推送信号给关联的 BPAgent |

### 目标 3.2：客户上下文管理（Account Thread）

| 维度 | 说明 |
|------|------|
| **目标** | 维护这个客户的完整交互历史，任何人查看都能快速了解"这个客户是什么情况" |
| **输入** | 所有会议摘要（时间线）；所有邮件/聊天摘要；商机状态变更历史；关键事件（人员变动、预算周期、竞品出现） |
| **输出** | Account Thread 时间线（按时间排列的关键事件）；客户 360 度画像（行业、规模、需求、预算周期、决策流程）；数据覆盖率和新鲜度标注（哪些信息是最新的，哪些已过期） |
| **动作** | 持续聚合 → 自动归档旧事件 → 标注数据新鲜度 → 当 BPAgent 查询时提供结构化上下文 |

### 目标 3.3：跨商机协调

| 维度 | 说明 |
|------|------|
| **目标** | 一个客户下可能有多个商机，协调它们之间的关系和优先级 |
| **输入** | 该客户下所有商机 Agent 的状态和进展；客户整体预算信号；各商机的关键人重叠情况 |
| **输出** | 商机关系提示（"商机①和商机②共享决策人，推进节奏应协调"）；优先级建议（"商机①金额大但风险高，商机②金额小但确定性强"） |
| **动作** | 监控所有子商机状态 → 发现冲突或协同机会 → 推送建议给关联的 BPAgent |

## 4. 商机 Agent（⊂ 客户 Agent）

**简介**：一个 Deal 的"专属管家"。挂在客户 Agent 下，专注于这一个商机从立项到成交（或流失）的全过程管理。

### 触发时机

| 触发方式 | 触发条件 | 说明 |
|---------|---------|------|
| 被动 | 客户 Agent 转发新会议事实 | 会议中与本商机相关的讨论 → 重新评估阶段/风险/预测 |
| 被动 | 销售 BPAgent apply 阶段变更 | 销售确认推进 → 更新阶段并回写客户 Agent |
| 定时主动 | 每日停滞检测 | 检查商机是否长时间无实质推进 → 超阈值则预警 |
| 定时主动 | 定期风险重算 | 综合多因子重新评估风险评分 |
| 被动 | 关键人参与度变化 | 决策人缺席会议、联系人变动等信号 |

### 目标 4.1：商机阶段推进

| 维度 | 说明 |
|------|------|
| **目标** | 跟踪商机当前阶段，识别推进条件是否满足，建议下一步 |
| **输入** | 最新会议中与该商机相关的讨论（来自销售 BPAgent apply 的素材）；阶段定义和推进标准；当前已满足/未满足的条件 |
| **输出** | 阶段状态卡片（当前阶段 + 已满足条件 + 缺失条件）；阶段推进建议（"建议从'需求确认'推进到'方案评估'，因为…"）；推进阻碍提示（"还差'预算确认'和'技术验证通过'两个条件"） |
| **动作** | 接收新事实 → 评估阶段条件 → 如果满足 → 生成推进建议 → 通过通讯层推送给 BPAgent → 销售 confirm → apply 阶段变更 → 回写状态到客户 Agent |

### 目标 4.2：风险识别与预警

| 维度 | 说明 |
|------|------|
| **目标** | 持续监控风险信号，在问题恶化前发出预警 |
| **输入** | 商机停滞时间（多少天没有实质推进）；关键人参与度变化（决策人是否还在参加会议）；竞品信号（会议或聊天中提到竞品）；预算/时间线变化信号 |
| **输出** | 风险评分（高/中/低 + 具体因子）；风险预警卡片（"商机停滞14天，决策人最近2次会议未出席"）；降风险建议（"建议尝试联系技术侧的张工，绕过当前阻塞"） |
| **动作** | 持续评估风险因子 → 超过阈值 → 推送预警给销售 BPAgent → 如果风险持续升高 → 同时推送给主管 BPAgent（触发升级） |

### 目标 4.3：预测与下一步建议

| 维度 | 说明 |
|------|------|
| **目标** | 预测赢单概率，推荐最有效的下一步动作 |
| **输入** | 商机完整时间线；类似商机的历史成交模式（mock 阶段为规则匹配）；当前阶段的典型下一步模板；客户 Agent 提供的客户上下文 |
| **输出** | 赢单预测（概率 + 置信度 + 关键影响因子）；下一步建议卡片（"建议在本周内完成技术方案演示，历史上在这个阶段超过10天未演示的商机赢率下降40%"）；关键里程碑提醒 |
| **动作** | 定期重新计算预测 → 推送下一步建议 → 销售可追问原因 → 销售 confirm 下一步 → 创建为执行任务 |

### 目标 4.4：商机信息回写

| 维度 | 说明 |
|------|------|
| **目标** | 将商机维度的变化同步回客户 Agent，保持上下文一致 |
| **输入** | 商机阶段变更；金额调整；关键人变化；赢单/丢单结果 |
| **输出** | 状态变更事件（写入客户 Agent 的 Account Thread）；如果赢单 → 客户健康度上调信号；如果丢单 → 丢单原因分析 |
| **动作** | 商机状态变化 → 自动回写客户 Agent → 客户 Agent 更新 Account Thread 和健康度 |

## 核心场景：会后流程（Post-Meeting Flow）

> 流程图见 `docs/diagrams/post-meeting-flow.png`

销售参加完客户X的会议后，完整的 Agent 协作流程：

| 步骤 | 执行者 | 动作 | 产出 |
|------|--------|------|------|
| ① | 销售（人） | 发送会议链接给自己的 BPAgent | 原始素材进入系统 |
| ② | 销售 BPAgent | 接收素材，分类识别（客户X · 商机①） | 素材分类结果 |
| ③ | 销售 BPAgent | 调用模型理解会议内容 | — |
| ④ | 销售 BPAgent | 生成会议摘要卡片 | 关键议题 · 决策 · 参会人态度 · 关键信号 · 建议下一步 |
| ⑤ | 销售（人） | 查看摘要，可追问细节 | — |
| ⑥ | 销售（人） | confirm 摘要准确 | — |
| ⑦ | 销售 BPAgent | apply 到客户X Agent | 会议事实写入客户上下文 |
| ⑧ | 客户X Agent | 更新 Account Thread · 重算健康度 · 检测关系变化 | 客户健康度变化信号 |
| ⑨ | 商机① Agent | 评估阶段条件 · 重算风险 · 重算赢单预测 | 阶段推进建议 · 风险变化 · 下一步建议 |
| ⑩ | 通讯层 | 聚合为结构化卡片推送给销售 | 阶段推进卡片 · 风险提醒 · 下一步建议 · 健康度变化 |
| ⑪ | 销售（人） | 查看新卡片，追问，confirm 下一步 | — |
| ⑫ | 销售 BPAgent | apply → 可选 sync CRM | 商机状态更新 |
| ⑬ | 商机① Agent | 回写状态到客户X Agent | Account Thread 更新 |
| ⑭ | 商机① Agent | 如风险升高 → 通知主管 BPAgent | 风险升级事件 |
| ⑮ | 主管 BPAgent | 推送辅导/关注卡片给主管 | 主管决策 |

**关键观察**：
- 销售只做了一个动作（发送会议链接），后续 3 个 Agent 自动协作
- 销售在两个关键节点做 confirm：摘要确认（⑥）和下一步确认（⑪）
- 风险升级自动触发主管 BPAgent，无需销售手动上报
- 整个闭环遵守 suggestion → confirm → apply → sync

## 层间数据流总结

```
① 数据层（Raw Input）
   会议录音 · 聊天 · 邮件 · CRM · 文档
         │
         ▼
② Agent 层
   ┌─ 销售 BPAgent ──素材理解──→ 客户 Agent ──状态更新──→ 商机 Agent
   │       ↑                          │                       │
   │       │←── 事实+建议 ────────────┘                       │
   │       │←── 风险+预测+下一步 ─────────────────────────────┘
   │       │
   │  主管 BPAgent ←── 上报/汇总 ──── 销售 BPAgent ×N
   │       │──── 辅导/审批 ──────→ 销售 BPAgent ×N
   │       │──── 审批/介入 ──────→ 商机 Agent
   │
         ▼
③ 通讯层
   对话引擎 ↔ 卡片路由 → 主动推送
         │
         ▼
④ 前端展示层
   / 对话主界面 · /pipeline · /meetings/:id · /deals/:id · /customers/:id
```

## 关键设计原则

1. **BPAgent 面向人，客户/商机 Agent 面向对象**：人通过 BPAgent 交互，客户/商机 Agent 不直接面向人类用户
2. **会议是核心事实源**：所有 Agent 的判断最终都要追溯到会议、邮件、聊天等原始素材
3. **卡片是唯一交互介质**：Agent 之间、Agent 与人之间的交互都通过结构化卡片完成
4. **suggestion → confirm → apply → sync 不可合并**：任何 Agent 输出都是建议，必须人确认后才能生效
5. **Explainability 强制**：每个 Agent 输出必须可追溯到证据，标注置信度和数据新鲜度
